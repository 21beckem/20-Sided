  Note: You'll need a build tool (Vite or Rollup) to compile Svelte components.
  The build tool creates bundles for each page's JS file.

  Key Design Decisions

  Frontend Organization (Multi-Page Application with Svelte)

  - pages/ - Multiple HTML pages (not a single-page app)
    * Each page is a separate HTML file with its own URL
    * Traditional navigation between pages using links/redirects
    * Example: /index.html, /map-editor.html, /session.html

  - components/ - Svelte components (.svelte files)
    * Reusable UI components like Navbar, MapBuilder, MapGrid
    * Each component encapsulates its own template, logic, and styles
    * Imported and mounted by page-specific JS files

  - js/ - Page initialization scripts
    * Each HTML page has a corresponding JS file
    * Mounts Svelte components into the page
    * Handles page-specific logic and interactions
    * Example: map-editor.js imports and mounts MapBuilder.svelte

  - services/ - Centralized API communication layer
    * Shared across all pages for consistency
    * Handles authentication, data fetching, error handling

  - utils/ - Shared utility functions
    * DOM helpers, validators, local storage management

  Backend Organization

  - MVC Pattern - Models, Routes, Controllers separated for maintainability
  - config/ - Centralized configuration management
  - middleware/ - Authentication, validation, and error handling
  - utils/ - Shared backend utilities

  How Multi-Page + Svelte Works

  1. Each HTML page links to its corresponding JS file
     Example in map-editor.html:
     <script type="module" src="../js/map-editor.js"></script>

  2. The JS file imports and mounts Svelte components
     Example in map-editor.js:
     import MapBuilder from '../components/MapBuilder.svelte';
     new MapBuilder({ target: document.getElementById('app') });

  3. Svelte components are built/bundled using a build tool (Vite/Rollup)
  4. Navigation happens via traditional HTML links (<a href="my-maps.html">)

  Benefits of This Structure

  1. Separation of Concerns - Clear boundaries between frontend/backend
  2. Scalability - Easy to add new features without cluttering
  3. Maintainability - Logical organization makes code easy to find
  4. Testing - Dedicated test directory with unit/integration tests
  5. Team Collaboration - Clear structure helps team members know where to add code
  6. SEO Friendly - Multiple HTML pages are easier for search engines
  7. Simpler Routing - No need for client-side routing library
  8. Progressive Enhancement - Pages work without JavaScript, enhanced with Svelte

  Frontend (devDependencies):
  - svelte
  - vite (or rollup with @rollup/plugin-node-resolve, @rollup/plugin-commonjs)
  - @sveltejs/vite-plugin-svelte (if using Vite)
  - rollup-plugin-svelte (if using Rollup)

  Backend (dependencies):
  - express
  - dotenv
  - bcrypt or bcryptjs (password hashing)

  - cors
